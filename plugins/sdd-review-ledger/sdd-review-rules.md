# SDD Review Rules (项目可选附加规则)

> **怎么生效**：把本文件放在 **repo 根**（或用 `SDD_REVIEW_RULES_FILE` 指向任意路径）。
> 命中后，它的内容会被**限长（≤4KB / 60 行）+ 净化**，作为「项目附加规则」段注入
> **每回合首条完整提醒**——让你的团队规则真正进入模型上下文，而不必指望模型自己去 Read。
> **不放**此文件时，插件仍内置一套评审协议（提醒里的 4 步取证 + 最终门槛照常生效）；
> 本文件是在其之上的**附加**项，**不是替换**——删它或留空都只是少了这段附加规则。
> 本文件**不参与**清除判定（清除唯一信号永远是勾选 `.sdd-review-todo.md`）。

## 你的角色

你是这次变更的**唯一语义裁判**。工具只机械地告诉你"哪些文件自上次评审后变了"，
**判断它们是否真的需要同步、归到哪个 change-dir、要不要改文档**——全部是你的事。
工具从不替你下"已对齐 / 有偏差"的结论。

## 最终回复前必检（硬门槛，勿略）

1. 若本轮 review 中你编辑过 code / design / tasks → **立刻重读最新 `.sdd-review-todo.md`**。
2. 其中**新出现的 `path@hash`** 也要逐项评审并勾选（编辑文件不自动清除）。
3. 待评审区**不为空**时，不要说"SDD review 已完成"，要说明还剩哪些待人工确认。

> `pending` 不等于"有 bug"，它表示"这一版文件还没被你或人确认过"；doc-only 的 pending 可能是
> 有意保留的计划（等后续代码或人工确认）。但**别在嘴上说完成、却把它留在待评审区**。

## 评审纪律（先取证，后结论）

对每一个待评审项，**先产出事实，再下结论**，不接受裸判断：

1. 相关 design/tasks 此刻**声称**什么（引用具体一句/一段）。
2. 对应 code 此刻**实现**什么（引用具体函数/行为）。
3. 二者是否一致——指出冲突点，或明确写"经对照无冲突"。
4. 结论：
   - **需改** → 编辑对应 `design.md` / `tasks.md`，让**文档**与 code 现状一致（评审是同步
     文档，不是改实现）。若 design/tasks 描述了 code 尚未实现的功能，那是**计划而非缺陷**——
     不要在评审里替用户补写实现代码，实现是用户单独发起的步骤。
   - **无需改**（纯重构 / 格式化 / 与该 change-dir 无关）→ 在 `.sdd-review-todo.md`
     勾掉该项，理由里带上第 3 步的依据。

## 归属评审规则

- 一个 code 变更可能与某个 change-dir 相关、也可能与全部都无关。候选 change-dir
  已列在提示里——**由你判断**它属于哪个，或判为 `unrelated`。
- 反向（design/tasks 领先 code）：这是**计划，不是实现命令**。文档跑在代码前面只会被记进
  `.sdd-review-todo.md`，**不会**主动催你评审；评审时也**不要**为了"对齐"去补写 code 尚未
  实现的部分——实现由用户单独发起。你只需确认文档本身是你想表达的，必要时在待评审项里留一句
  "code 尚未实现 X"。

## 清除（唯一信号 = 勾选）

- 评审完每一项，**都要**在 `.sdd-review-todo.md` 把它勾掉（`[ ]` → `[x]`），
  即使你的结论是"无需改"。**编辑文件不会自动清除**——这是刻意的（防止"碰一下文档
  就误清掉一条没人审过的评审项"）。
- 勾选只能发生在 `## 待评审` 区域：原地把该行 `[ ]` 改成 `[x]`，保留原始
  `path@hash`，并在行尾追加一句证据理由。不要把条目移动到"审计历史"区，不要手写新增
  待评审条目。
- 勾选时保留行内的 `@<hash>`：它把你的结论钉在你当时看过的那一版内容上；内容再变会
  自动重新变成待评审。
- 如果评审过程中你又编辑了 code / design / tasks，先重新读取最新 `.sdd-review-todo.md`，
  再勾选最新出现的 `path@hash`。不要只勾旧 hash。

## 边界

- 纯格式化 / lint / gofmt 触发的待评审项：一句"仅格式化，无需改文档"勾掉即可（良性）。
- 拿不准就**留着不勾**：留在待评审清单里是安全的，后续相关编辑会继续提醒，不会静默丢弃。
