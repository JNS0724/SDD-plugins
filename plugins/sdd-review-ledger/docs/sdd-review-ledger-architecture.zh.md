# SDD Review-Ledger 架构（初始设计方案）

**版本:** 0.1（初稿 / clean-slate）
**日期:** 2026-05-30
**作者:** Claude（Opus 4.8）+ 用户协同
**状态:** 全新项目的初始设计。**不继承** `sdd-drift-check` 的任何实现包袱；PRD、mtime 状态机、link-lock 事件溯源方案全部作废，仅保留经多代理对抗评审验证过的结论。
**血脉:** 本方案是对 `docs/sdd-drift-check/sdd-drift-check-link-lock-architecture.zh.md` §19 第三轮评审结论 + 后续"能否实现核心诉求"反思（4 路独立设计者 + 对抗压测一致收敛）的落地。lineage 见 §11。

## 一、需求内核（剥到一句话）

> **vibe coding 时，自动化辅助评审 文档↔文档、代码↔文档 之间的偏差。**

派生约束（从"vibe coding"推导，凡与之冲突的设计一律否决）：

- **零纪律**：用户不做规范化 SDD。任何要求"手维护锚点 / 每次显式 align 盖章 / 配 CI / 手维护 checked-in lockfile"的方案，违背前提。
- **辅助评审（assist review）**，不是"工具自己算出有没有偏差"。重点在 assist。
- **跨会话** / **子代理代劳** / **双平台（Claude Code + OpenCode）** 都算。
- **永不阻断**用户主流程（fail-open）。

---

## 二、第一性原理：为什么不"机械检测偏差"

**"代码↔文档偏差"本质是语义判断，没有确定性算法。** 没有机械方法能知道 `greet.ts` 是否仍符合 `design.md#greeting`。

前两代方案都死在同一个元错误上——**让确定性工具去回答语义问题**：

| 方案 | 代理信号 | 真面目 |
|---|---|---|
| 当前实现 | file mtime + session 时序 | 变更检测 |
| link-lock | 内容哈希 + 显式 align + 事件重放 | 变更检测 |

两者本质都只是**变更检测（change detection）**，区分不了三类"内容变了"：

| 变更类型 | 应否提醒评审 | 工具能机械区分吗 |
|---|---|---|
| 真语义漂移（代码行为变了，文档没跟） | 应 | ❌ |
| 无行为重构 / 格式化 | 不应 | ❌ |
| 文档润色 / 改错别字 | 不应 | ❌ |

这条鸿沟只能由**语义裁判**填。强行让工具填 → 要么噪音（误报），要么橡皮图章（漏报）。link-lock 三处 P0 致命缺陷（跨会话塌缩无确定性规则、两分支 align 合并歧义、捕获非 100% 丢归属）**同根**：都来自"让工具用事件重放 COMPUTE 偏差"。

**关键洞见**：vibe coding 时，回路里**已经坐着唯一够格的语义裁判——agent 本身**。所以工具根本不该去"算偏差"，而应做"评审编排 + 记账"：可靠地在变更后把材料端到 agent 面前请它评审，并记住它的裁决。

> 此结论由 4 个独立设计者（极简 / 机械-语义边界 / LLM 唯一裁判 / 诚实合约 四种切入点）+ 1 个种子假设，**全部独立收敛**得出；对抗压测后 5 个设计无一致命。这是该问题的自然解，而非某个聪明设计。

## 三、诚实合约（本方案的核心边界）

把工具从「偏差检测器」重新定义为「**评审编排器 / 评审账本**」。

### 工具保证（机械、可靠、无 LLM）

1. 用**内容哈希**（非 mtime）记录哪些代码/文档变了。"需评审" = 当前哈希 ≠ 账本里上次已评审哈希（或从未评审）。这是**真·纯函数**，无事件重放、无因果推断、无顺序依赖。
2. 在**编辑发生的那次工具结果上**把"这些文件改了 + 相关 design/tasks 路径 + 评审规则"端给 agent；并永远落一份**人可见**的 `.sdd-review-todo.md`（不依赖任何 hook 投递成功）。
3. 跨会话、子代理透明、永不阻断。

### 工具委托（判断只活在这一处）

- "是不是真漂移 / 文档要不要改 / 归属哪个 change-dir" → **全交 agent**。账本只记下它的 verdict（盖在它当时看过的确切哈希上，内容再变即失效重提）。

### 工具坦白做不到（任何方案都跨不过的硬底）

- **不保证"代码真符合 design"**。评审质量 = LLM 质量 = vibe coding 本身的质量上限。工具只保证"评审被可靠端到裁判面前并留痕"，不保证"判得对"。

### 对外措辞纪律（防止橡皮图章错觉）

工具对外**只会说两种话**：
- 「这对 `path:hash` 还没被评审过」
- 「这对被评审为 X、且内容自评审后未变」

它**从不**说「无偏差 / 已对齐 / 检测到漂移」。把"声称"与"已验证事实"的区分写进措辞本身。

## 四、两层架构

把核心诉求拆成两半——它们的实现难度差异巨大：

```
┌──────────────────────────────────────────────────────────────┐
│  Layer A  doc↔doc（design.md ↔ tasks.md，同 change-dir）        │
│    纯工作树函数：随时可重算，无需事件捕获、无需投递、平台无关       │
│    触发 = 任一文档自上次"相互评审基线"后内容哈希变了              │
└──────────────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────────────┐
│  Layer B  doc↔code（代码 ↔ design/tasks）                      │
│    账本 + 会话内投递 + LLM 裁判                                 │
│    触发 = 代码文件当前哈希 ∉ 任何覆盖它的 verdict               │
│    归属（哪个 change-dir）= LLM 现场判（候选都列给它）           │
└──────────────────────────────────────────────────────────────┘
       共享：哑账本（path→reviewedHash+verdict）、LLM 裁判、
             编辑时工具结果投递 + .sdd-review-todo.md 兜底
```

**Layer A 是白送的简化**：同 dir 的 design ↔ tasks 偏差，纯靠"当前工作树哈希 vs 账本基线"即可触发评审提示，**零事件捕获、零投递依赖**，在 OpenCode 上也 100% 能算出触发。只有 Layer B 才需要会话内捕获 + 投递。

> 注意：两层的「触发」都是机械的；两层的「判断」都仍交给 LLM。Layer A 只是不需要捕获 edit 事件就能算出"该提醒了"。

## 五、数据模型：哑账本

**设计原则：账本越哑越好。** 不存"干净/脏"派生真相，不跑状态机，不做 deriveLock。

```jsonc
// 项目级账本。每个被追踪元素一条记录：
{
  "src/greet.ts": {
    "reviewedHash": "sha256:a1b2c3",      // LLM 上次评审时看到的内容哈希
    "verdict": "synced",                   // synced | no-change-needed | unrelated（仅展示，工具不据此做状态机）
    "reviewedAt": "2026-05-30T10:00:00Z",
    "by": "agent"                          // provenance：声称者
  }
}
```

- **"需评审"判定**（纯函数）：`当前工作树哈希(element) ≠ record.reviewedHash`（或无记录）→ 进待评审清单。这是"有没有被评审过"（**可机械判定**），不是"有没有偏差"（**不可机械判定**）。
- **verdict 失效**：record 钉死它覆盖的确切哈希。内容一变即自动失效、重新提出评审。**不需要 age/TTL 鲜度治理**——过期是机械可判的。
- **自愈**：账本缺失/损坏 → 删掉即可（最坏情况：把当前工作树重新判成"需评审"，多一轮评审，无数据可腐烂）。

### 5.2 格式、持久化与并发（§13#1 已定）

**格式 = 单个 JSON map（`path → record`）。** 不用 JSONL/append/内容寻址/union/compaction——那套（§19 已判 YAGNI）唯一的真收益"无锁可并集合并"只在账本经历 **git 三方合并** 时才兑现，而 v0.1 账本不进 git。账本本质是 `last-write-per-key` 的 map，把它序列化成 append 事件流等于把已删掉的状态机从持久化层偷渡回来。

**持久化 = 本地 + gitignore（账本与 todo 同时忽略）。**
- 位置：`<nearest .git>/sdd-review-ledger-state/ledger.json`；兜底 `<cwd>/.sdd-review-ledger-state/`，再兜底 `%TEMP%/`。只放账本，无任何派生快照。
- **硬规则——绝不"半 committed"**：`ledger.json` 与 `.sdd-review-todo.md` 的 git track 状态**必须一致**。只 commit todo 不 commit 账本 → 队友本机空账本 → 全判未评审、把 todo 里的 `[x]` 冲回、原作者 ack 丢失。`init` 同时把两行写进 `.gitignore`；工具启动检测 `todo tracked XOR ledger tracked` → 告警（不阻断）。
- 跨人/跨 clone/换机共享 = **v0.2 团队模式**（届时两文件一起 commit、账本升 JSONL+内容寻址并集、补"同元素冲突 verdict = 重评审"归并，绝不静默择一）。v0.1 自愿放弃，诚实标注（§10#5）。

**并发与原子性**（同机多会话/子代理是独立进程，纯进程级竞争、不经 git）：
- 写账本全程持 `ledger.json.lock`（沿用旧实现 `locks.js`：O_EXCL + 有界重试 + stale-mtime 反捕防死锁）。
- 原子落盘：tmp + `renameSync`（同目录）。**砍掉旧实现"rename 失败 → 非原子 writeFileSync"的降级分支**——rename 失败就保留 tmp、本轮放弃写。
- 拿不到锁 / 写失败 → **fail-open 跳过本轮写**（§1 永不阻断 > 必须写成功）。
- **关键：丢一次写在本架构里失败安全。** verdict 是哈希键化的 ack；丢掉一条 ack = 该项不被清除 = 重新浮出 = 良性重评审，**绝不会**把未评审项错标已评审（false-clean）。所以 JSON map 整文件 RMW 在并发缝隙（锁超时丢写 / stale 偷锁双写覆盖）下最坏只丢 ack（→ 重审），不产生有害结论——这正是选它而非 JSONL 的依据：JSONL 的"绝不丢写"优化的是本架构**不需要**的属性（§8 良性 ≪ 有害），却背回 §19 的 YAGNI 复杂度。
- 损坏自愈：解析失败 → 当空账本（删重建）→ 最坏重判一轮，无可腐烂数据。

### 5.3 todo 输入+输出协调：ingest-before-render + 哈希钉选

`.sdd-review-todo.md` 既是输入（勾选 = ack）又是输出（渲染待评审清单）。单次运行固定序（全程同一把锁、同一次读盘的 todo）：

1. **ACQUIRE** ledger 锁（拿不到 → fail-open，仅纯展示重算，ingest 推迟下轮）。
2. **INGEST（必须先于 render，防丢勾选命门）**：解析磁盘 todo 的结构化 `[x] <path>@<hash>`（只认勾选框 + 路径 + 内联哈希，不解析 NL 理由）。每个新勾选 → 写一条 verdict（no-change/unrelated）**覆盖该条目内联的那个哈希**。
3. **COMPUTE** `computeNeedsReview(工作树, 账本)`。
4. **RENDER** 用 (账本 + 工作树) 重写 todo。
5. **WRITE + RELEASE** 原子替换 ledger 与 todo，释放锁。

- **为什么 ingest 必须先于 render**：先 render 会用旧账本覆盖 todo，把刚勾的框冲回 `[ ]` → 丢勾选。
- **为什么 ack 钉"内联哈希"而非"ingest 时当前哈希"**（堵一个真 false-clean）：人勾框到工具 ingest 之间文件可能又变。若按当前哈希记 verdict = 给"没人审过的新版本"盖章 = 跨版本 false-clean。故 todo 条目内联它所指哈希（§8.6 契约 `<path>@<short-hash>`），ack 只覆盖那个哈希；勾完又改 → 新哈希 ∉ verdict → 正确重新 pending。
- 勾选本身是一次 write 事件 → 即时触发本流程；漏投也无妨（todo 持久，下轮 ingest 必捞到 `[x]`）→ 最终一致。
- 人手改 todo 容错：删行 ≠ ack（哈希仍 ≠ 基线则下轮加回）；畸形行跳过；`[x]`→`[ ]` 无 verdict 写入。

## 六、机械层：捕获 / 触发 / 投递

### 6.1 捕获（确定性、无 LLM）

- 写工具事件（Edit/Write/MultiEdit）后，对目标文件算内容哈希，分类 code / SDD-doc（纯路径/扩展名判断）。
- **不存 mtime、不存 session 内存、不做时序推断。** `git checkout` / 不改字节的操作不影响哈希。

### 6.2 触发（纯函数，任何触发点同一答案）

- Layer A：扫各 change-dir，`design.md` 或 `tasks.md` 当前哈希 ≠ 账本基线 → 该 dir 进待评审。
- Layer B：代码文件当前哈希 ∉ 任何覆盖它的 verdict → 进待评审。
- 可由 hook / CLI / pre-commit 任意处跑出**相同结果**（纯函数 of 工作树 + 账本）。子代理漏 hook 也能被后续任意一次重算抓到"哈希变了"。

### 6.3 投递（要害，见 §7.1）

- **主通道 = 编辑那次工具结果**（Claude Code = PostToolUse；OpenCode = `tool.execute.after`）。提醒搭"编辑本身"的车，**编辑时就送达**，不依赖任何后续工具调用，也不依赖不可靠的 Stop。
- **兜底 = `.sdd-review-todo.md`**（repo 根，人可见）：即便所有 hook 投递失败，用户打开仓库就能看见。这是不依赖投递成功的唯一可靠原语。**它同时是 §8.2 的 ack 入口**（勾选清除），故是一等工件，非纯展示。
- **跨会话兜底 = 下一轮 UserPromptSubmit / chat.message** carry-over。
- 提醒沿用结构化 `<system-reminder>[SYSTEM DIRECTIVE]` 形态（这是当前实现已验证可用的模型可见契约）。
- **投递 payload 含完整 git diff**（行号免费且精确）：让 LLM 无需猜就能定位"这次 doc/code 变了哪几行、影响哪些对手件"，把整文件粒度（§6.4）"损失的段定位"由 LLM+diff 免费补回。

### 6.4 obligation 锚定模型（§13#3 整文件粒度已定 → 推论）

锚点粒度已定为**整文件哈希**（理由见 §13#3）。由它推出的精确 obligation 规则——既防 N-fold 噪音、又不漏报反向 drift：

- **obligation 锚在"哈希漂了的那个元素"上，一次编辑只生一条。** 编辑 design.md → 仅 design.md 这一个元素哈希漂 → 挂一条「design.md 变了，评审下游」obligation。**绝不在记账时枚举受影响的 N 个对手件**——那等于工具预先断言"这 N 个都可能 drift"，正是 §2 否决的"让工具回答语义问题"，且整文件粒度下 doc 动一字就 N 条全重开 = N-fold 噪音。
- **受影响对手件由 LLM 在投递时用 git diff 现场圈定**（§3 委托 / §4 归属交 agent）。"doc 里哪一段相关"是 LLM + git diff 免费且擅长的活，工具不重做。
- **Layer B 的 code obligation 由 code 自身哈希变更独立触发**，与 doc 编辑解耦 → 一次 doc 编辑不会重开 N 条 code obligation。
- **反向 drift（design 改了、code 可能落后）不漏报**，靠"一条 doc obligation + LLM"而非"重开 N 条 code"：design.md 哈希漂 → 该 dir 进待评审 → 投递候选清单**含该 dir 关联的 code 文件** → LLM 判 code 是否需跟进。注意：code↔doc verdict 的**重开触发只认 code 侧哈希**（不按 `{code,doc}` 对键化），否则任何 doc 编辑都会 N-fold 重开 code——反向 drift 改由这条单 doc obligation + LLM 兜底。
- **残余（诚实标注，§8.5 同级）——跨段 false-clean**：整文件把同一 doc 的多个关注点塌缩成一个可清除单元。若 design.md 同时改了 A、B 两个关注点，agent 只评审/同步了 A 就记 verdict（覆盖整文件新哈希）→ B 的 design↔tasks 漂移被**静默连带清除**。缓解：投递必含完整 diff（让 agent 看见 A、B 都变了；评审全部变更段是 agent 职责 = §10#1 LLM 完整性上限）。硬化（verdict 只覆盖 LLM 本回合显式点名的对手件、不刷新整 doc 基线）列为 **v0.2 项，且明确不引入段级锚点纪律**。

## 七、三个设计要害（必须避开的陷阱）

这三条是对抗压测从失败设计里挖出的"血泪"，是本方案能否成立的关键。

### 7.1 主投递是"编辑时工具结果"，不是 Stop

OpenCode 上 Stop continuation 不可靠（实测）。每次 Edit 本身就是一次 `tool.execute.after`——把提醒挂它身上，编辑时即送达。**绝不能**把主投递押在 Stop 上（那是 link-lock / 部分旧设计的退化点）。

### 7.2 绝不能让工具从"观察到 agent 改了 / 读了 doc"自动合成 verdict

这是 §19.3.1 / P0-1 的复活陷阱：若工具看到 agent 编辑了某 doc 锚点就自动盖"已解决"，那么 agent 为改个错别字碰一下 doc，就会**静默清掉一条真实的、无人裁决过的 code↔doc 评审义务** → false-clean 漏报。

**规则：清除必须显式（仅勾选），编辑从不自动清除。**（2026-05-30 定，解 §8.2 ↔ §6.4 旧冲突）
- ✅ **唯一清除 = 显式勾选 `[x] <path>@<hash>`**（§8.2）。对 code/doc 元素一视同仁，由 agent/人 主动声明"这个元素在此哈希评审过"。
- ✅ **agent 编辑文件去同步**是评审里的真活，但它**只改该元素哈希、不构成清除**——编辑完仍需对评审过的各元素显式勾选（理由见 §8.2：编辑 doc 同时是"产生下游评审 obligation"的来源，不能同时充当清除）。
- ❌ **"Read 一眼 = 已评审"必须砍掉**——最隐蔽的 false-clean（agent 常为找上下文而 Read）。
- ❌ **绝不从"观察到 agent 编辑了 doc"自动盖 code 的 verdict**——code 是否真匹配新 doc 是语义判断，只能由 agent 对该 code 显式勾选声明（堵 §19.3.1 跨 path auto-synth）。

### 7.3 不要在工具层做"归一化 / AST 去噪"

为了抑制格式化噪声而在工具里做语义归一化，等于把刚外包出去的 COMPUTE 又偷渡回来，并重新引入"什么算 no-op"的判断。**保持账本哑**：格式化触发就触发，让 LLM 一句话否掉（"仅格式化，无需改文档"）。误报在此被降级为**良性误触发**（多看一眼），而非有害的错误结论。归一化只作为纯可选的 token 优化，永不进核心。

## 八、清除信号：已定（v0.1）

**结论先行**：采用「**显式清除 + 被动留存**」——清除一条待评审项只认两种**显式、机械可观测**的动作；绝不从 Read 或无关编辑隐式反推。

### 8.1 "三难"其实是假三难

原表的"永不清除 → 反复 nag 疲劳"这一角，只在把两个本应分开的投递面混为一谈时才尖锐。拆开即化解：

| 投递面 | 行为 | 化解的 horn |
|---|---|---|
| **主动 agent 提醒**（搭编辑时工具结果） | **节流 / 有界**（每会话 / 每批次上限） | agent 不被反复轰炸 |
| **被动人可见 `.sdd-review-todo.md`** | 未清除项**无限留存**，但被动（想看才看，从不 nag） | 未评审项不被静默丢弃 |

于是"不自动清除"不再等于"永远 nag"，而是"安静留在被动清单上，直到被显式 ack"。这是**安全失败模式**：既不 false-clean，也不无限 nag。剩下的只是"显式信号长什么样"，不再是"清不清"。

### 8.2 清除模型（唯一信号 = 勾选；编辑只产生 obligation、从不清除）

`.sdd-review-todo.md` 升级为**一等工件**：既是人可见兜底，又是 agent/人 的 ack 入口（机器只读其结构化"勾选框 + 路径 + 内联哈希"，**不解析 NL**）。

**唯一清除信号 = 勾选 `[x] <path>@<hash>`。** 对 code 与 doc 元素一视同仁：勾掉某元素 = "该元素在此哈希被显式评审过"，按该哈希记一条 verdict（`synced` / `no-change` / `unrelated` 仅作展示标签，一行理由原样存 provenance）。
- 走**可靠写通道**（编辑 todo = 一次 write 工具事件 = `tool.execute.after`），不押 OpenCode 上不可靠的 Stop / transcript。
- **统一人与 agent**：人消项也是勾同一个框。

**编辑文件从不自动清除任何 obligation。** 编辑只改该元素哈希——若 ≠ 账本基线则它（重新）"需评审"，直到被显式勾选。这条是刻意的（2026-05-30 定，解 §8.2 ↔ §6.4 旧冲突）：
- **必须如此**：否则"编辑 design 自动清 design"会让"design 改了 → 评审 tasks/code 是否落后"这条反向 drift 永不触发——编辑 doc 正是**产生**下游评审 obligation 的来源，不能同时充当它的清除。
- **堵 false-clean（§7.2）**：工具绝不从"观察到 agent 编辑了 doc"推断"那条 code 已同步"；code 是否真匹配新 doc 是语义判断，只能由 agent 对该 code 显式勾选。

**典型流**：code C 改 → 提示"评审 design" → agent 看 diff，需要则编辑 design（design 自身随之变成"需评审下游"）→ 在**一次 todo 编辑**里勾掉它评审过的所有元素（`C@hash`、`design@hash`、必要时 `tasks@hash`），各带一行理由。多一次勾选，但 agent 做、廉价、且无 false-clean。

### 8.3 明确否决

- ❌ **"Read 一眼 = 已评审"**：最隐蔽的 false-clean（agent 常为找上下文而 Read）。
- ❌ **"两次 Stop 没动作 = 接受"**（旧 PRD J11 的 two-strike）：OpenCode 上"agent 没动作"与"agent 根本没收到提醒"不可区分（投递 best-effort）→ 会变成 false-clean。
- ❌ **跨 dir 把一次 doc 编辑绑定到"某条具体 code obligation"**：多 dir 时无确定性依据（link-lock §19.3.1）。

### 8.4 哈希键化 → 自动重开

任何 verdict（synced / no-change / unrelated）都钉死它覆盖的确切哈希。被勾掉的项若内容再变 → 新哈希 ∉ verdict → **自动重新变"待评审"**。热文件自然 churn（其 todo 条目被内容变更不断顶替），冷的未 ack 项正确地长期留存。**无需 TTL / age 鲜度治理**——过期机械可判。

### 8.5 残余（诚实标注）

- agent 始终不 ack 也不改 doc → 项**留在 todo**（安全：工具尽到"端出 + 留存"，人能看到一摞待评审）。严格优于旧实现的静默漏报。
- ~~"agent 为无关理由编辑了某 doc 会把 code 记成 synced（窄 false-clean）"~~ → **已消除**：2026-05-30 定 checkoff-only（§8.2），编辑从不清除 code，code 只能被对其自身的显式勾选清除，故此 false-clean 路径不存在。代价转为"多一次勾选"（良性摩擦）而非"漏报"。

### 8.6 `.sdd-review-todo.md` 格式契约（v0.1）

```markdown
## 待评审
- [ ] <path>@<short-hash>  (候选: <change-dir>[, <change-dir>...])

## 已评审
- [x] <path>@<short-hash> — <一行理由>
```

工具只解析：**勾选框状态 `[ ]`/`[x]` + 路径 + 内联 `@<short-hash>`**（结构化）。`@<short-hash>` 是 ack 的钉选锚——勾选记的 verdict 覆盖**这个哈希**而非 ingest 时的当前哈希（防跨版本 false-clean，见 §5.3）。理由为自由文本，原样存为 provenance，**不解析语义**。该格式做字节级 snapshot 测试（类比旧实现对 prompt 模板的 snapshot），确保修订不静默改变契约。

## 九、关键场景走查

| 场景 | 行为 |
|---|---|
| 单会话完整流（写 design/tasks→改代码） | 改代码那次工具结果即提示"代码改了，相关 design/tasks 在此，评审是否需同步"；agent 同步 doc 或声明无需 → 记 verdict。**无静默漏报，无硬阻断。** |
| 子代理改代码 | 子代理编辑即使 hook 未注册，下次主代理侧任意触发重算工作树哈希即发现"≠ 账本" → 提示评审。不依赖捕获子代理事件。 |
| 跨会话改代码（SDD 来自历史） | 账本项目级持久 → 新会话首个触发点即浮出"未评审变更"。 |
| 只改 tasks.md（design 落后） | Layer A 纯函数：tasks 哈希 ≠ 基线、design 未同步 → 提示"design 可能落后，请评审"。**平台无关、无需捕获。** |
| 纯格式化 / gofmt 全仓 | 哈希变 → 触发评审 → LLM 一句话否掉（良性误触发，非错误结论）。 |
| DTS / hotfix 无关改动 | 进待评审 → LLM 判 unrelated → 记 verdict（unrelated）。不再静默跳过整轮，但也只是一次轻量裁决。 |
| 存量 repo 冷启动 | `init`：把当前工作树哈希全部记为 baseline、**不触发评审**（避免满屏"全未评审"）。这是一次静默 baseline，但不声称"已验证"，只表示"不对历史存量回溯追责"，伤害小。 |
| 会话崩溃 / 投递丢失 | 账本在 repo/状态目录不丢；`.sdd-review-todo.md` 人可见；下一轮/下次触发重新浮出。 |

## 十、诚实底线：残余风险（任何方案都跨不过或需收口）

1. **LLM 评审质量**（不可约）：`no-change-needed` 可能看走眼。工具只留痕不纠错，= vibe 质量上限。可加"无理由 / no-change 比例过高"告警，但治标。
2. **OpenCode 投递 best-effort**：主通道 `tool.execute.after` 已是最可靠通道，但平台仍可能不投递；shell 重定向写、未注册 hook 的子代理编辑**捕获不到**。缓解：编辑时投递（不押后续工具）+ 工作树重算补抓"变了" + 人可见 todo。**真正不可恢复的只剩"工具从未看见的写"**——但此时任何方案都无能为力，且未来任意一次重算 + todo 会最终抓到。
3. **误触发成本**：高频小编辑 / 格式化下重复提示有 token 与打扰成本。靠 §7.3 良性化 + 编辑时单次 + 节流缓解；不靠工具层归一化。
4. **冷启动 baseline 是一次静默 ack**：诚实标注为"不回溯存量"，非"已验证"。
5. **跨会话共享依赖账本持久**：不 commit 时退化为同机有效（不比旧实现差，但也未超越）。
6. **跨段 false-clean**（整文件粒度的代价，见 §6.4）：同一 doc 多关注点塌缩成一个可清除单元，评审 A 段可静默连带清掉未评审的 B 段。缓解=投递含完整 diff；硬化为 v0.2 项（不引入段级锚点纪律）。
7. **ack 延迟一轮**（账本格式的代价，见 §5.3）：锁争用 fail-open 或勾选落在临界区后 → ack 延迟一轮浮现（todo 持久不丢、失败安全）。需在 todo 头注"勾选下次运行生效"管理预期。
8. **网络/同步盘**（NFS/Dropbox/SMB）：`renameSync`/O_EXCL 原子性不保证 → 账本可能损坏；靠 §5.2 自愈（损坏当空账本重判一轮）兜底，不额外加机制。

## 十二、最小原型验收（证伪/证实本方案）

核心赌注：**"需评审 = (工作树, 账本) 的纯函数"，且把判断完全外包给 LLM，能在 vibe coding 下可靠辅助评审而不静默漏报、不有害误报。**

1. `computeNeedsReview(工作树, 账本)`：纯函数，Layer A（doc↔doc）+ Layer B（doc↔code）各跑通；同一输入任意触发点结果一致。
2. `recordVerdict(element, hash, verdict)`：显式裁决写账本；**绝不从观察 edit/read 自动合成**（§7.2）。
3. 编辑时工具结果投递 + `.sdd-review-todo.md` 兜底；OpenCode 上验证主通道是 `tool.execute.after`、不押 Stop。
4. **误报对账（一等验收，对应需求"不误报"）**：构造一组真实编辑序列——多关注点文件改无关函数、全仓 gofmt、design 段润色/重组——断言这些"无语义漂移"的改动**不产生有害错误结论**（只允许良性"请瞥一眼"提示，且能被一句话消解）；并与旧 mtime 方案在同序列上的误报次数对账。
5. 跨会话：账本持久，新会话首触发点浮出历史未评审变更。
6. 冷启动：`init` 不把存量全判成"需评审"。

> 不"盲目实现"，也不"分析瘫痪"。原型先证伪核心赌注与三个要害（§7），再考虑铺开。

## 十三、开放问题（下一轮收口）

1. ~~**账本持久化格式**~~ → **已定（JSON map + 本地 gitignore）**，见 §5.2/§5.3。3 路论证：格式实下放于"账本是否经历 git 三方合并"；v0.1 不进 git → JSON map + FsLock + 原子 rename 足够，JSONL/union/compaction 是 §19 已判 YAGNI。关键依据：本架构里"丢一次写"失败安全（丢 ack→重评审，绝不 false-clean），故 JSON map 的并发弱点只换来良性重审、不换来有害结论；JSONL 的"绝不丢写"优化的是本架构不需要的属性。采纳对抗评审的两点硬化：① ack 钉"内联哈希"（§5.3/§8.6，堵跨版本 false-clean）；② 砍非原子写降级分支。checked-in + JSONL = v0.2 团队模式（两文件一起 commit）。
